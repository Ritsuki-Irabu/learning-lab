# X投稿文生成スキル

learning-labの学習ログを参照し、X投稿文の骨格を2〜3候補生成する。
一言解釈はユーザーが書く。生成後にフラグを立てるか確認する。

---

## フロー

### 1. 学習ログを取得

```bash
cd /home/claude/learning-lab && git pull origin main
cat README.md
```

### 2. 候補ログを選定

以下の優先順位で対象ログを選ぶ：

1. **今日の日付**のログ（未フラグのもの）
2. 今日のログがない、または全てフラグ済みの場合 → **過去ログの未フラグ**からピックアップ

**重複チェック（必須）：**
直近3件のフラグ済みログの技術名を確認し、同じ技術名が連続しないように選ぶ。

**未フラグの判定：**
学習ログテーブルの「参照」列に `✅` がないものを未フラグとみなす。

### 3. 投稿文の骨格を2〜3候補生成

**文字数制限：投稿本文（ハッシュタグ含む）を 200文字以内 に収める。**

X の仕様上、日本語は1文字=2、ASCII は1文字=1としてカウントされ上限は280。
ただし一言解釈はユーザーが追記するため、骨格生成時点で **200文字以内** を目安とする。

**文字数カウント方法：**
- 日本語・全角記号・絵文字 → 各2としてカウント
- 英数字・半角記号・スペース → 各1としてカウント
- 改行 → 各1としてカウント

生成後、各候補の末尾に概算文字数を表示する：
```
（約 XXX / 280文字）
```

各候補を以下フォーマットで提示する：

```
【候補1】
絵文字 本文（学習内容をぼやき混じりの口語で整理）
→ 一言解釈（←ここだけあなたが書いてください）

#エンジニア学習 #任意タグ

（約 XXX / 280文字）
```

## 投稿の方向性（現場エンジニア視点）

目指すのは「現場のエンジニアが本当に呟きそうな温度感」。note記事やQiita要約ではなく、タイムラインでスクロールの手が止まる**ぼやき混じりの学びメモ**。

### 文体の温度感

- **教科書感を出さない**:【ラベル】形式や「いかがでしたか」系のまとめは使わない
- **完璧に整えすぎない**: 「ちょっと崩れてる」くらいが自然。語尾を「〜だな」「〜けど」で締めるなど口語混じりOK
- **絵文字は分野インデックスとして1つだけ**: 🧪テスト / 🗄️DB / ⚛️React / 🐍Python など固定運用。本文中への散らしはしない
- **AI感を出さない**: 箇条書きの過剰使用、絵文字の散らし、過度な構造化を避ける

### 「本音/感想」の軸合わせ

本文の主題と、最後に添える本音の軸を必ず揃える。ズレるとちぐはぐになる。

| 本文の主題 | 本音を向ける先 |
|---|---|
| ツールの使い分け | ツールへのぼやき（「一つに揃えてほしい」など） |
| 概念・原則 | 概念の理想と現実のギャップ（「頭では分かってるけど現場だと…」） |
| 学んだ気づき | 過去の自分への突っ込み（「上から書きたくなるけど下を固めた方が結局速い」） |

### リアルな感想の引き出し集

状況に応じて使い分ける言い回しのストック：

- 「頭では分かってるけど、実践するの難しい」
- 「理想は分かるけど、現場だとついE2Eに頼りがち」
- 「上から書きたくなるけど、下を固めた方が結局速いんだよな」
- 「気をつけてても上が膨らみがち」
- 「一つに揃えてほしいけど」（ツール愚痴系）
- 「分かってからが本番」
- 「過去の自分に教えたい」
- 「結局これに戻ってくる」
- 「やってみて初めて腹落ちした」

### 誠実さの担保

- **言い切らない**: 一般論として書くと「それ違くない?」とツッコまれる。「自分のスタックでは」「うちの構成だと」「目安として」を添える
- **出典のある数字には出典を**: 「70:20:10(by Google)」のように添えて知ったかぶり感を消す
- **分かってる人ワードを1つ忍ばせる**: 「アイスクリームコーン」「N+1」「YAGNI」など、知ってる人がニヤッとする用語を主張せず入れる

### ハッシュタグのルール

- 基本タグ: `#エンジニア学習` 固定
- **最大2個まで**（3個以上はリーチが落ちる傾向）
- 分野タグは冒頭の絵文字インデックスで代替できるので、無理に足さない

### 4. フラグ確認

候補提示後、以下を聞く：

```
どの候補でフラグを立てますか？（1 / 2 / 3 / 他を見たい）
```

- 番号を選んだ → 該当ログにフラグを立てて完了
- 「他を見たい」→ 別の候補を提示して3に戻る

### 5. フラグを立てる

選ばれた候補の学習ログ該当行の「参照」列に `✅` を追記する。

**変更前：**
```
| 2026-05-03 | Docker | Engine起動不具合 | [メモ](docs/docker/README.md) |
```

**変更後：**
```
| 2026-05-03 | Docker | Engine起動不具合 | [メモ](docs/docker/README.md) ✅ |
```

フラグ追記後、mainへ直接コミット＆プッシュする：

```bash
git add README.md
git commit -m "Post: X投稿フラグを立てる（<技術名>）"
git push origin main
```

## 注意事項

- 「内容」はコピペしてそのまま使えるように簡潔に整理する
- 「一言解釈」の欄は絶対に埋めない（ユーザーの言葉が必要）
- 完成度を求めない・理解途中の内容でもOK
- プッシュにはPATが必要な場合がある（認証エラーが出たらユーザーにPATを求める）

## 用語の置き換えルール

文字数削減のため、正式名称ではなく一般的なカタカナ・日本語表現を使う。
エンジニアっぽさは残しつつ、難読な正式名称は噛み砕く。

| 正式名称 | 置き換え例 |
|---|---|
| Docker Engine | エンジン |
| npipe / 名前付きパイプ | パイプ |
| Engine running | エンジン起動 |
| Docker Desktop UI | UI |
| コンテナランタイム | コンテナの中身 |

コマンド（`wsl --shutdown` など）はそのまま使う。
半角スペースはそのまま維持する。
