# Learning Lab ドキュメント管理スキル

技術資産ドキュメントを **正確に・深く思考して** 作成・更新する。
単なるテンプレート埋めではなく、技術的な正しさを検証した上で記録する。

---

## 基本方針

- **正確性優先**：曖昧な記述より「わからない」と明示する方が良い
- **ソース意識**：公式ドキュメント・実際の動作に基づいて記述する
- **エラーを見つけたら必ず訂正**：誤った情報はリポジトリの信頼性を損なう
- **フォーマット統一**：全ドキュメントをJava READMEの構造に合わせる

---

## ドキュメント標準フォーマット

以下の構造を全技術ドキュメントで統一すること：

```
# <技術名> 技術資産ドキュメント

> <技術名>を用いた学習・開発経験を体系化した技術資産。
> 学習内容の再利用性向上・設計思考ログの蓄積・ポートフォリオ提示を目的とする。

## 技術スタック       ← 使用言語・FW・ツールのテーブル
## 学習プロセス       ← 学習スタイルと到達レベルのテーブル
## チートシート       ← 実際に動作確認済みのコードスニペット
## 実装資産           ← 作ったものの一覧・ディレクトリ構成
## 技術的課題と改善方針 ← 課題・問題点・改善方針のテーブル
## 今後の強化方針     ← 優先度別チェックリスト
## 学んだこと         ← 日付・トピック・メモのテーブル
```

## ワークフロー

### 1. 現状把握

対象ドキュメントを読み込み、以下を確認する：

```bash
cat docs/<技術名>/README.md
```

- セクション構成はフォーマットに合っているか
- コードスニペットに構文エラーや誤りがないか
- 技術的な説明が正確か（バージョン依存の情報に注意）
- 学習ログに記録漏れがないか

### 2. 技術的正確性の検証

チートシートや説明文の各項目について以下の観点で思考する：

**コードスニペットの検証チェックリスト：**
- [ ] 構文が正しいか（括弧・セミコロン・型など）
- [ ] APIやメソッド名が現在のバージョンで有効か
- [ ] コメントの説明と実際の挙動が一致しているか
- [ ] より良い書き方・推奨パターンがあれば反映しているか

**説明文の検証チェックリスト：**
- [ ] 技術用語の使い方が正確か
- [ ] 因果関係・仕組みの説明が正しいか
- [ ] 古い情報・非推奨の情報が含まれていないか

### 3. 誤りの訂正

検証で見つけた問題を**必ず訂正**する。訂正時は：

- 何が間違っていたか
- なぜ間違いだったか
- 正しい内容は何か

を学んだことテーブルに記録することを検討する。

### 4. コンテンツの充実

空欄・プレースホルダー（`- | -` や `<!-- ... -->`）が残っている箇所は、
実態に合った内容で埋める。推測で埋めるより「学習中」「未着手」と明示する方が良い。

### 5. フォーマット統一

標準フォーマットに沿っているか確認し、不足セクションを追加する。
既存の内容を削除せず、再配置・補完する。

### 6. コミット・プッシュ・PR作成

CLAUDE.md のワークフローに従う：

```bash
# ブランチ作成（まだなければ）
git checkout -b docs/<技術名>-update

# 変更をコミット
git add docs/<技術名>/README.md
git commit -m "docs: <技術名>ドキュメントを更新・誤り訂正"

# プッシュ
git push -u origin <ブランチ名>
```

その後 `mcp__github__create_pull_request` でPRを作成する（base: main）。

## 訂正記録の書き方

誤りを訂正した場合は学んだことテーブルに以下の形式で記録する：

| 日付 | トピック | メモ |
|---|---|---|
| YYYY-MM-DD | [訂正] <トピック名> | 誤：<旧内容> → 正：<新内容> |

---

## 注意事項

- コードスニペットは「動くはず」ではなく「動作確認済み」のものを優先する
- バージョンに依存する情報は対象バージョンを明記する
- 確信が持てない内容には `> ⚠️ 要確認：` の注記を付ける
- 誤りを見つけても文脈が不明な場合はユーザーに確認してから訂正する
