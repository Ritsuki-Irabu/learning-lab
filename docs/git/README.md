## チートシート

### 基本ワークフロー

```bash
# ブランチ作成・切り替え
git checkout -b feat/login

# 変更をステージング・コミット
git add <file>
git commit -m "feat: ログイン機能を追加"

# リモートへプッシュ
git push -u origin feat/login
```

### ブランチ一覧確認

```bash
git branch -a        # ローカル＋リモート
git branch -r        # リモートのみ
```

### マージ

```bash
# staging へマージ（PR経由が原則）
git checkout staging
git merge --no-ff feat/login
git push origin staging
```

## 補足：実務ブランチ戦略の全体像

```
main（本番環境）
 └── staging（ステージング環境）
      ├── feat/login
      ├── feat/user-profile
      └── fix/cart-bug
```

### ブランチの役割

| ブランチ | 対応環境 | 説明 |
|---|---|---|
| `main` | 本番（Production） | リリース済みの安定コード。直接コミット禁止 |
| `staging` | ステージング（Staging） | リリース前の検証環境。`main` へのマージ前に QA を行う |
| `feat/*` / `fix/*` | ローカル・開発 | 機能・修正単位のブランチ。`staging` へ PR を出す |

### Feature Branch を切る判断基準

**ブランチを切る（分ける）条件：**
単独で `staging` にマージしても、`staging` が正常に動作する場合。

例：
- 新しい API エンドポイントの追加（フロントエンドとは独立して動く）
- UI コンポーネントの追加（既存画面に影響を与えない）

**ブランチを切らない（まとめる）条件：**
片方だけでは動作しない・意味をなさない変更の場合。

例：
- DB カラム追加のマイグレーション ＋ そのカラムを使うモデル変更（どちらか片方だけでは整合しない）
- API エンドポイント追加 ＋ そのエンドポイントを呼ぶフロントエンド処理（API だけ出しても機能しない場合）

> ⚠️ 「単独でマージしても staging が壊れないか」を基準に判断する。壊れるなら1ブランチにまとめる。
